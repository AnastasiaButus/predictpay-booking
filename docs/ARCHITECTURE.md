# PredictPay BookingGuard Architecture

PredictPay BookingGuard is an educational production-like ML service for
predicting hotel booking cancellation risk. The ML model is intentionally simple
compared with the service architecture around it: a REST API, durable database,
async worker queue, credit billing, dashboard, observability, and repeatable
Docker Compose demo flow.

## High-Level Architecture

```text
Streamlit dashboard
        |
        | REST + JWT
        v
FastAPI backend ---- PostgreSQL
        |
        | Celery task id / primitive prediction_id
        v
Redis broker ---- Celery worker ---- trusted sklearn Pipeline artifact
        |
        v
Flower monitoring

FastAPI also exposes /metrics for Prometheus-compatible scraping.
```

## Components

- FastAPI: API layer for health, auth, users, billing, promocodes, predictions,
  OpenAPI docs, request logging, and `/metrics`.
- PostgreSQL: source of truth for users, refresh tokens, model metadata,
  predictions, transactions, promocodes, and promocode activations.
- Redis: Celery broker/result backend for async prediction execution.
- Celery worker: consumes prediction jobs from `default` and `priority` queues
  and runs inference against DB-backed prediction records.
- Streamlit: Russian-language dashboard that talks to the backend only through
  REST API calls.
- Flower: local/demo Celery monitoring UI at http://localhost:5555.
- Prometheus `/metrics`: text endpoint with application, HTTP, and prediction
  submission counters/histograms.

## Data Flows

### Auth

1. User registers or logs in through FastAPI.
2. Passwords are hashed before storage.
3. Access JWT is returned to the client.
4. Refresh token is stored in PostgreSQL as a SHA-256 hash, not in plaintext.
5. Protected endpoints resolve the current user from the access token and DB.

### Billing

1. User balance lives in PostgreSQL on the `users` table.
2. Top-up is a mocked payment operation that creates a completed `top_up`
   transaction.
3. Prediction submit reserves credits before the Celery task is queued.
4. Worker success confirms the charge.
5. Worker failure refunds the reserved credits.

### Promocode Activation

1. User submits a code or the secret challenge URL through the API/dashboard.
2. Backend validates promocode activity, expiry, activation limits, and duplicate
   user activation.
3. Successful activation increments balance and writes a `promo_bonus`
   transaction.
4. The Poincare-style challenge validates URL format only in the MVP.

### Async Prediction Lifecycle

1. `POST /api/v1/predictions` validates the 12-feature request contract.
2. Backend checks the active prediction limit before creating a prediction or
   reserving credits.
3. Active statuses are `pending` and `processing`.
4. Free users are limited to 3 active predictions.
5. Pro users and admins are limited to 10 active predictions.
6. Backend creates a `pending` prediction, reserves credits, and enqueues a
   Celery task with only `prediction_id` as the task argument.
7. Free users route to the `default` queue; pro/admin users route to `priority`.
8. Worker loads the prediction from PostgreSQL, marks it `processing`, runs
   inference, then writes `completed` or `failed`.
9. Clients poll `GET /api/v1/predictions/{id}` or
   `GET /api/v1/predictions/history`.

## Billing Consistency

The billing lifecycle is:

```text
reserve -> charge/refund
```

- Submit: `balance` decreases and `reserved_balance` increases.
- Success: `reserved_balance` decreases and a zero-amount
  `prediction_charge` transaction records the confirmed charge.
- Failure: `reserved_balance` decreases, `balance` increases, and a
  `prediction_refund` transaction records the refund.

PostgreSQL remains the source of truth for balances and prediction status. The
DB/broker boundary is MVP-level and is not a production-grade distributed
transaction.

## ML

- The predictor loads a trusted local sklearn Pipeline joblib artifact.
- User-uploaded model artifacts are not supported.
- The feature contract has exactly 12 features:
  `hotel`, `lead_time`, `adults`, `children`, `previous_cancellations`,
  `booking_changes`, `deposit_type`, `customer_type`, `market_segment`,
  `required_car_parking_spaces`, `total_of_special_requests`, and `adr`.
- Leakage columns are excluded and rejected:
  `reservation_status`, `reservation_status_date`.
- Risk labels are derived from cancellation probability:
  `low < 0.35`, `medium < 0.65`, `high >= 0.65`.

## Observability

- Backend logs are structured JSON on stdout.
- Request logging includes method, path template, status, duration, client host,
  and `request_id`.
- `X-Request-ID` is preserved when supplied, otherwise generated by the backend.
- Request bodies, passwords, JWTs, refresh tokens, and Authorization headers are
  not logged.
- `/metrics` exposes Prometheus-compatible metrics:
  `predictpay_http_requests_total`,
  `predictpay_http_request_duration_seconds`,
  `predictpay_predictions_submitted_total`, and `predictpay_app_info`.
- Flower gives local visibility into Celery workers and tasks.

## Limitations And Future Improvements

- No user-uploaded models.
- No real payment gateway.
- No S3/MinIO object storage in the MVP.
- Flower has no authentication in local/demo mode.
- No production-grade distributed transaction between PostgreSQL and Redis/Celery.
- Synthetic dataset fallback is only a smoke-check for demo reproducibility.
- The real Kaggle CSV is local-only and ignored by git.
